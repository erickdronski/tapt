# Tapt — Data model notes (sellable taste-graph + territory intelligence)

## Event-first taste-graph fields (the atomic sellable unit = one check-in)
- checkin_event.id uuid PK
- checkin_event.user_id uuid FK -> user_profile (personal plane join key; never in exports)
- checkin_event.beer_id uuid FK -> beer_catalog
- checkin_event.brewery_id uuid FK -> brewery
- checkin_event.sku_canonical_id uuid FK -> sku_taxonomy (proprietary join key to CPG systems)
- checkin_event.style text + substyle text
- checkin_event.abv numeric(4,2)
- checkin_event.ibu smallint
- checkin_event.srm smallint
- checkin_event.rating numeric(2,1) check 0-5
- checkin_event.flavor_tags text[] (hoppy/malty/sour/fruity/roasty...)
- checkin_event.geo_bucket_h3 text (H3 res-8/9 cell; NO raw lat/long in sellable layer)
- checkin_event.venue_id uuid FK -> venue
- checkin_event.on_off_premise on_off_premise (enum: on_premise, off_premise)
- checkin_event.occasion occasion_kind (enum: home, bar, restaurant, event, sports)
- checkin_event.event_ts timestamptz + generated day_of_week smallint, daypart daypart_kind, season season_kind
- checkin_event.price_paid numeric(6,2) + price_tier price_tier_kind
- checkin_event.purchase_intent_flags jsonb (saved, wishlisted, where_to_buy_clicks)
- checkin_event.consent_version text + sale_optin bool + location_optin bool + gpc_flag bool (row-level consent provenance)
- checkin_review.aspect_scores jsonb (ABSA appearance/aroma/palate/taste/overall; personal plane; raw review_text stays on-device)
- taste_vector.user_id uuid PK + top_styles text[] + abv_comfort_band numrange + ibu_comfort_band int4range + novelty_score real + price_tier_pref price_tier_kind (derived; personal plane only, never sold)

## Schema additions (Supabase / Postgres + PostGIS, Lore house style)
- ENUM TYPES (mirror Lore house style): create type on_off_premise as enum ('on_premise','off_premise'); create type occasion_kind as enum ('home','bar','restaurant','event','sports','other'); create type daypart_kind as enum ('morning','afternoon','evening','late_night'); create type season_kind as enum ('winter','spring','summer','fall'); create type price_tier_kind as enum ('value','mid','premium','luxury'); create type consent_purpose as enum ('functional','analytics_aggregate','sale_share','precise_location'); create type consent_action as enum ('grant','withdraw','gpc_optout','dns_optout');
- TABLE sku_taxonomy (the proprietary moat): id uuid PK, canonical_name text, brand text, sub_brand text, style text, substyle text, abv numeric(4,2), ibu smallint, srm smallint, apple_map_item_id text null, created_at/updated_at timestamptz + touch trigger. This is the join key to CPG systems - competitors must rebuild it.
- TABLE brewery: id uuid PK, name text, sku_taxonomy_brand_ref, apple_map_item_id text null (MKMapItem.identifier for the Apple join), home_geo_bucket_h3 text, verified_partner bool default false, created_at/updated_at + touch trigger.
- TABLE venue: id uuid PK, name text, apple_map_item_id text null, geo_bucket_h3 text not null, on_off_premise on_off_premise, poi_category text (Apple MKPointOfInterestCategory), lat/lng numeric (PERSONAL PLANE precision, RLS-locked, never exported), created_at/updated_at + touch trigger.
- TABLE checkin_event (event-first spine, immutable): id uuid PK, user_id uuid FK -> user_profile on delete cascade, beer_id uuid FK -> beer_catalog, brewery_id uuid FK -> brewery, sku_canonical_id uuid FK -> sku_taxonomy, style text, substyle text, abv numeric(4,2), ibu smallint, srm smallint, rating numeric(2,1) check (rating between 0 and 5), flavor_tags text[], geo_bucket_h3 text not null, venue_id uuid FK -> venue, on_off_premise on_off_premise, occasion occasion_kind, price_paid numeric(6,2), price_tier price_tier_kind, purchase_intent_flags jsonb default '{}', event_ts timestamptz not null default now(), day_of_week smallint generated always as (extract(dow from event_ts)) stored, daypart daypart_kind, season season_kind, consent_version text not null, sale_optin bool not null default false, location_optin bool not null default false, gpc_flag bool not null default false, created_at timestamptz default now(). RLS: owner-only read/write on the personal plane.
- TABLE checkin_review (personal plane; raw text never leaves device-derived): checkin_id uuid FK -> checkin_event, user_id uuid, aspect_scores jsonb (ABSA), flavor_ontology_tags text[], sentiment_confidence real, created_at. RLS owner-only.
- TABLE taste_vector (derived, personal plane, never sold): user_id uuid PK -> user_profile on delete cascade, top_styles text[], abv_comfort_band numrange, ibu_comfort_band int4range, novelty_score real, price_tier_pref price_tier_kind, refreshed_at timestamptz. Service-role maintained.
- TABLE consent_ledger (the gating asset - durable, exportable, append-only): id uuid PK, user_id uuid FK -> user_profile on delete cascade, purpose consent_purpose not null, action consent_action not null, granted bool not null, policy_version text not null, ui_text_shown text not null, source text (e.g. 'onboarding_toggle','ATT_prompt','GPC_signal'), region_code text, created_at timestamptz not null default now(). NO update/delete (immutable audit trail); indexed on (user_id, purpose, created_at desc); an Edge Function exports the current-state view per user for M&A diligence and AB 1824.
- TABLE sensitive_location_suppression: id uuid PK, geo_bucket_h3 text, category text (addiction_treatment/medical/reproductive/religious/shelter), source text, created_at. Queried at aggregation time to EXCLUDE these geopoints from any sellable roll-up.
- MATERIALIZED VIEW / TABLE aggregate_cell (the ONLY thing sold): geo_bucket text, style text, window_start/window_end, distinct_users int check (distinct_users >= 10), distinct_venues int, checkin_count int, style_share numeric, momentum numeric, flavor_sentiment_score numeric, dp_noise_applied bool. Populated by a service-role Edge Function that enforces k-anonymity (k>=10 users AND min venues), minimum-sample suppression (auto-roll-up sparse geos/weeks), differential-privacy noise, sensitive-location exclusion, and drops any row where a contributing user has a matching sale_share opt-out or GPC flag in consent_ledger. No raw PII columns permitted.
- REGION SCOPING: add region_code / is_eu_user on user_profile so the EU dataset is architecturally separable and excludable from any data sale (GDPR). Add a lore-style security-definer helper (e.g. taste_sale_eligible(user_id)) so the aggregate pipeline reads consent state without tripping RLS recursion, mirroring the existing lore_trust_tier() pattern.

## Data-collection principles
- EVENT-FIRST, NOT PROFILE-FIRST: the atomic sellable unit is one immutable check-in row per consumption moment (the Untappd/Vivino 'consumption moment' asset), never a per-user favorites list. Collect from record one; retrofitting a churned base is impossible.
- Per check-in event, capture: beer_id + brewery_id + sku_canonical_id (mapped to a PROPRIETARY canonical brand/brewery/style taxonomy that is itself a core moat), style/substyle, abv, ibu, srm, rating 0-5, structured flavor_tags[] + ABSA aspect_scores (appearance/aroma/palate/taste), geo_bucket (H3/ZIP3/DMA - never raw lat/long in the sellable layer), venue_id, on_off_premise flag, event_ts with derived day_of_week/daypart/season, occasion (home/bar/restaurant/event/sports), and purchase-intent (saved, wishlisted, where_to_buy_clicks, price_paid, price_tier).
- INSTRUMENT THE CPG SIGNALS OTHER APPS SKIP: daypart/season, on- vs off-premise, occasion/context, and pre-purchase intent. Circana/NIQ already own POS scanner data - what they PAY for is this attitudinal + occasion + intent layer they cannot derive from till data.
- Derive per-user preference vectors (top_styles[], abv/ibu comfort bands, novelty_score, price_tier_pref) to power the recommendation flywheel, but keep these on the PERSONAL PLANE - never sold at the individual level.
- TWO-PLANE ARCHITECTURE WITH A HARD BOUNDARY. Personal plane (never sold): raw check-ins, exact venue, timestamp, friends, review_text - user-controlled; coarse flavor-tag NLP runs on-device and only anonymized tags leave the phone. Aggregate plane (the only thing sold): server roll-ups keyed geo-bucket x style x rolling-window.
- Every event row carries consent metadata (consent_version, sale_optin, location_optin, gpc_flag) so the aggregation pipeline can EXCLUDE any opted-out user BEFORE roll-up - this is what makes the aggregate layer defensible and AB 1824-safe.
- Optimize for per-market DENSITY over vanity national DAU: saturate 2-4 launch metros so you can credibly tell a brewery/distributor 'we have statistically meaningful volume in YOUR territory.' Density x exclusivity x contracted B2B revenue x refresh velocity are the valuation levers - not raw DAU.
- Maintain a sensitive-location SUPPRESSION LIST from day one (addiction-treatment/AA sites, medical, reproductive, religious, shelters) so those geopoints NEVER enter the sellable layer - the exact FTC hot-button for an alcohol app. Truncate/round precision and cap retention.
- Aggregate gates at query time: k-anonymity k>=10 distinct users AND a minimum distinct-venue count, minimum-sample suppression (sparse geos/weeks auto-roll-up), and calibrated differential-privacy noise on published counts/shares.
- Two-stage flavor/sentiment NLP: Stage 1 fine-tuned encoder does aspect-based sentiment analysis (BeerAdvocate 4-aspect schema) on-device; Stage 2 server-side LLM pass handles negation/sarcasm/slang on already-de-identified text. Aggregate sentiment only at geo x style x window under the same k-threshold.

## Territory intelligence (the B2B / sellable layer)
- THE MARKET IS PROVEN, NOT HYPOTHETICAL: Untappd Insights already sells this exact territory asset (style/flavor trend monitoring, social sentiment, location-based consumption) as 'aggregated, anonymized, privacy compliant' off 10M+ users and 1B+ consumption moments; in 2025 users checked into 1.5M unique beverages across 2.6M locations. This is a competitive product spec, not a green-field bet.
- CONSUMER FEATURE = geo-native discovery, map as the home surface: Local Leaderboards (adjustable radius), 'What's Hot Near You' (ranks by 7/30-day check-in VELOCITY and rating-trend, not all-time volume, so it surfaces genuinely rising local beers instead of re-showing national hype everywhere), Regional Taste Maps (choropleth over metro/DMA/ZIP3/H3 with a style toggle), Trending Styles by City (week-over-week style-share), and Brewery Popularity by Area (local reputation decoupled from national scale).
- NAMED, CITABLE INDICES published with sample size n and window (sub-threshold cells suppressed): Style Demand Index (local share indexed to national=100), Momentum (period-over-period delta - the 'up 30%' figure), Flavor Sentiment Score (ABSA net positivity per attribute per region), Local Favorite Rank (rating x local velocity, PII-safe).
- 2026 DEMAND SIGNALS ARE SALABLE NOW: hazy IPA palate fatigue + West Coast IPA revival, Session Hazy/Hazy Pale growth (4-5.5% ABV), broad lager resurgence, N/A beer ~33.7% YoY on-premise growth; U.S. craft production -4% to ~22.0M bbl in 2025 even as volume share ticked to 13.4%, with hyperlocal small breweries the growth pocket.
- TWO-STAGE NLP over the BeerAdvocate 4-aspect schema (appearance/aroma/palate/taste + overall): Stage 1 fine-tuned encoder does aspect-based sentiment analysis tagged to a beer flavor ontology (malt/hop/yeast/wild/off-flavor) runnable on-device; Stage 2 server-side LLM pass handles negation/sarcasm/slang normalization on de-identified text. Aggregate sentiment only at geo x style x window.
- PRIVACY IS A SOLVED ENGINEERING PROBLEM AND A FEATURE TO SELL: k-anonymity k>=10 is the accepted floor (k=10 cut max re-identification risk from 100% to ~1.09%), plus a minimum distinct-venue count so one dominant bar can't be de-anonymized, minimum-sample suppression that auto-rolls-up sparse geos/windows, and differential-privacy noise on published aggregates - exactly as 2025 CPG data clean rooms do. This lets the product answer 'what's hot in Austin' without ever exposing that a specific user drinks a specific beer at a specific bar.
- SAMPLE-BIAS GUARDRAIL: check-in users skew toward enthusiasts and hazy/hype styles, so 'regional demand' is demand among LOGGERS, not the whole market. Frame outputs as directional momentum/sentiment (not absolute market share), disclose the check-in basis, and calibrate against external anchors (Brewers Association production, distributor depletion) in clean-room engagements.
- INCUMBENT RISK: Untappd/Next Glass owns the category with 1B+ moments and existing brewery relationships; thin check-in volume produces noisy, easily-suppressed regional cells. Win a specific territory/niche DENSELY first (hyperlocal depth beats thin national breadth) before promising nationwide demand intelligence.

## Privacy & consent (so a future data-sale OR company sale is legally clean)
- THE COMPANY SALE (M&A) IS THE CLEAN PATH; the ongoing DATA SALE is the high-risk one - both are enabled by the same day-one disclosures. The existential risk is the FTC (not the App Store or CCPA): an alcohol app selling precise location is squarely in the 2024-2026 location-data-broker enforcement lane and the FTC has BANNED companies from selling sensitive location outright.
- Ship a counsel-drafted Privacy Policy BUSINESS-TRANSFER clause: personal data may be transferred as a business asset in a merger, acquisition, financing, reorganization, bankruptcy, or asset sale, and the successor may keep processing under this policy.
- Ship a ToS ASSIGNMENT clause: we may assign the agreement to a successor/acquirer WITHOUT user consent.
- DUAL DISCLOSURE in notice-at-collection: data may be SOLD/SHARED with third parties for stated purposes AND TRANSFERRED as an M&A asset - this dual disclosure keeps the company sale inside the CCPA business-transfer exemption and gives GDPR transparency cover.
- NEVER include an absolute 'we will never sell or transfer your data' line - it legally forecloses the exit and cannot be fixed retroactively on a churned user base. Purpose language must pre-authorize aggregated/anonymized third-party analytics + data-clean-room use.
- Layered, UNBUNDLED, default-OFF consent capture: neutral pre-permission explainer -> iOS When-In-Use location prompt (request Always only with a genuine background feature) -> separate opt-in toggles for 'share/sell my data with partners' and 'use my precise location for [named purpose]' -> the ATT system prompt with an honest NSUserTrackingUsageDescription. Selling to data brokers IS 'tracking' under Apple's definition, so ATT is mandatory; never gate features on ATT or bribe consent (5.1.2(i)).
- DURABLE CONSENT + OPT-OUT LEDGER from record one: timestamped, versioned, per-purpose, storing the exact UI text shown, EXPORTABLE to a buyer, including every CCPA do-not-sell/share and honored GPC signal. AB 1824 (in force Jan 1 2025) makes an acquirer inherit and honor every pre-close opt-out (penalties $2,500-$7,500/violation) - a clean exportable ledger is a diligence pass/fail that moves the price.
- Apple compliance: age-rate 18+ under the 2025 bands (17+ no longer exists) + first-launch DOB gate, keeping the DOB OUT of the saleable dataset. File Privacy Nutrition Labels declaring Location + Preferences/Purchases under BOTH 'Data Linked to You' AND 'Data Used to Track You'; ship a PrivacyInfo.xcprivacy manifest; audit every third-party SDK (their tracking is your violation).
- U.S. state law: publish 'Do Not Sell or Share' + 'Limit the Use of My Sensitive PI' links, honor GPC automatically, classify precise geolocation as SPI, ASSUME you are a California data broker (calendar Jan 1-31 CPPA registration ~$6,600, wire DROP bulk-deletion live Aug 1 2026 on a 45-day cycle, register in TX/OR/VT too), and run the CCPA risk assessment required before selling/sharing or processing SPI as of Jan 1 2026.
- GDPR/EU: selling data and monetizing location need freely-given, unbundled, withdrawable OPT-IN - legitimate interest will not hold; no tracking walls; fire no non-essential SDK before consent; GDPR anonymization must be IRREVERSIBLE (higher bar than CCPA de-identification). Keep the EU dataset ARCHITECTURALLY SEPARABLE so it can be excluded from a data sale a buyer isn't GDPR-comfortable with.
- DEFAULT COMMERCIAL PRODUCT = the aggregated, anonymized layer delivered via reports/dashboards/clean rooms, NEVER raw PII (the Untappd/Vivino model) - it minimizes 'sale of personal information' exposure, carries the highest margin, and is the format acquirers actually want.
